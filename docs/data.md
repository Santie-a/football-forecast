# Data

Distilled from roadmap.md Part 2, plus the artifact-store schema the app depends on.

## Target → source map

Each forecast target must come from a source that actually contains it.

| Target | National teams | Leagues |
|--------|----------------|---------|
| Scoreline / 1X2 | ✅ Intl results 1872–present (Kaggle) — results only | ✅ football-data.co.uk |
| Fouls / cards / corners | ⚠️ Only major tournaments, via StatsBomb open data | ✅ football-data.co.uk (incl. referee) |
| xG (a *feature*, not a target) | derive from StatsBomb events | FBref/Understat via `soccerdata` |

**Sequencing consequence:** scoreline + 1X2 for national teams first; add the
fouls/cards/corners count models only when moving to leagues, where the data to
fit *and validate* them exists.

## Sources (what each does and does not contain)

- **International results 1872–present** (Kaggle "International football results"):
  result, tournament, neutral-venue flag. **No** fouls/cards/corners.
- **StatsBomb open data** (`statsbombpy`): event-level (shots, fouls, cards with
  coordinates) for selected tournaments (World Cups, Copa América, AFCON, Women's
  WC). Lets you derive counts and your own xG. Free; mind terms of use.
- **football-data.co.uk**: free CSVs 2000/01–present — goals, shots, corners,
  **fouls, yellow/red cards, referee**, plus bookmaker odds. Main league source.
- **FBref / Understat** via `soccerdata`: richer league stats incl. xG. Rate
  limits + terms of use apply.
- **APIs (optional):** API-Football (detailed, freemium), football-data.org
  (thinner, free). Note rate caps.

## League data (Phase 4) — what we actually use

football-data.co.uk is the canonical league source but is **unreachable** from this
environment. We use a GitHub mirror, `xgabora/Club-Football-Match-Data-2000-2025`
(one `Matches.csv`, 42 leagues 2000–2025), via `data/sources/club_matches.py`
(decision 010). It carries goals, shots, fouls, corners, yellow/red cards, and
1X2 + over/under odds — but **no referee** (so M06 stays deferred). One division →
one processed parquet (`pipelines.ingest_league --division E0`).

## Leakage catalogue (M07)

Fields known only **after kickoff/full-time** — targets, never features for the
same match:

- **National-team source:** `home_goals`, `away_goals` (and the derived outcome).
- **League source:** `home_goals`/`away_goals`, `FTResult`/`HTResult`/`HT*`, shots
  (`HomeShots`/`AwayShots`/targets), `home_fouls`/`away_fouls`,
  `home_corners`/`away_corners`, `home_yellow`/`away_yellow`/`home_red`/`away_red`.

Safe (pre-kickoff): `date`, `competition`, `comp_type`, `home`, `away`, `neutral`,
`season`, bookmaker `odds_*` (used **only** as the benchmark, never as a model
input), and any pre-match Elo/rolling-form snapshot. Rule enforced in
`data/schema.py` and the no-leakage backtester.

## Recurring data tasks (do for every source)

- Understand each schema: column meanings, units, missing-data conventions, encodings.
- Locate the **referee** field — cards/fouls depend heavily on it.
- **Normalize team names** across sources (build a canonical mapping table; this
  is tedious and real — keep it in `src/football_forecast/data/`).
- Hunt for **leakage**: any field only known after kickoff/full time must never
  be a feature. Examples: full-time shots, post-match xG, final corner counts.
  Document each leak you find in `decisions.md`.

## Data handling rules

- `data/raw/` is immutable — exactly as downloaded, never hand-edited. All cleaning
  is code in `src/football_forecast/data/` so it is reproducible.
- `data/` and `artifacts/` are **gitignored**; they are regenerated by pipelines.
- Encode provenance: keep a note of source URL, download date, and licence per dataset.

## Artifact store schema

The contract between the PC (writer) and the Pi app (reader). Authoritative
definition lives in `src/football_forecast/store/`; keep this table in sync.

### `forecasts.sqlite` — table `forecasts` (sketch)

| column | type | meaning |
|--------|------|---------|
| `match_id` | TEXT | canonical fixture id |
| `date` | TEXT (ISO) | kickoff date |
| `home`, `away` | TEXT | canonical team names |
| `competition` | TEXT | tournament / league |
| `model` | TEXT | `elo`, `maher`, `dixon_coles`, `bayesian`, `boosting` |
| `market` | TEXT | `1x2`, `scoreline`, `over_under`, `correct_score`, `counts` |
| `payload` | JSON | market-specific (see shapes below) |
| `generated_at` | TEXT (ISO) | when the PC produced this row |

**Payload shapes (implemented).** One row per (match, model, market). Goal models
(maher, dixon_coles) publish the full bundle via `forecast/bundle.py`; result-only
models (elo) publish `1x2` only.

| market | payload |
|--------|---------|
| `1x2` | `{"H":.., "D":.., "A":..}` |
| `scoreline` | `{"goals": G, "matrix": [[P(i,j)]]}` — display slice 0..G |
| `over_under` | `{"1.5": {"over","under"}, "2.5": {...}, "3.5": {...}}` |
| `correct_score` | `{"i-j": p}` for the top scorelines |

### `backtests.parquet` (sketch)

`model × competition × split_end_date × metric` (RPS, log_loss, brier,
calibration_error), so the comparison page reads one file.

> Keep the app's queries to simple lookups/filters on these. If a new view needs
> a new shape, add a column/table on the PC side and re-sync — never make the app
> compute from raw data.
